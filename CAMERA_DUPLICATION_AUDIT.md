# Camera Intelligence Duplication Audit

## Scope

Audited camera-related data flow in `backend/analyzer.js` across:

- `deriveSocialCameraIntelligence()`
- `safeSocialCameraIntelligence()`
- `generateCameraLanguage()`
- `buildDirectorBrief()`
- `buildShotPlan()`
- `buildPromptSlots()`
- `buildStage2Context()`
- `compressStage2Assembly()`

No code was changed. This report identifies duplication candidates only.

## Executive Summary

Camera information is generated by two parallel systems:

1. `generateCameraLanguage()` produces grounded camera grammar from Stage 1 facts:
   - `framing`
   - `cameraMotion`
   - `cameraSpeed`
   - `lensBehavior`

2. `deriveSocialCameraIntelligence()` produces social-video camera intent:
   - `camera_style`
   - `camera_energy`
   - `camera_relationship`
   - `viewer_perspective`

The strongest duplication occurs when `buildDirectorBrief()` assigns:

```js
directorBrief.camera = socialCamera.camera_style || shotPlan.camera_behavior || cameraGrammar.cameraMotion || factual.camera_motion
```

Then `compressStage2Assembly()` serializes both:

- `DIRECTOR_BRIEF.camera`
- `COMPACT_CONTEXT.camera_style`

When social camera intelligence is present, these are usually the same value. This explains the repeated overlap diagnostic:

```txt
camera_style vs directorBrief.camera similarityEstimate = 1.0
```

The second major duplication is intentional but payload-expensive:

- `SHOT_PLAN.plan.camera_behavior`
- `PROMPT_SLOTS.populatedSlots.CAMERA_BEHAVIOR`

`PROMPT_SLOTS` is derived from `SHOT_PLAN`, so camera behavior is serialized twice in the final Stage 2 prompt.

## Camera Data Call Chain

```txt
Stage 1 facts
  -> camera_motion
  -> lens_feel
  -> camera_presence
  -> camera_engagement
  -> social camera fields persisted back onto factual object

Stage 2 platform generation
  -> safeSocialCameraIntelligence(factual)
       -> deriveSocialCameraIntelligence(factual)
       -> socialCameraIntelligence
       -> stage2Scope.socialCamera

  -> generateCameraLanguage(factual, "video")
       -> cameraGrammar

  -> buildShotPlan(factual, cameraGrammar, microMotion)
       -> shotPlan.camera_behavior

  -> buildPromptSlots(platform, shotPlan, factual, stage2Scope)
       -> PROMPT_SLOTS.CAMERA_BEHAVIOR

  -> buildDirectorBrief(factual, cameraGrammar, microMotion, shotPlan, stage2Scope)
       -> directorBrief.camera
       -> directorBrief.camera_style
       -> directorBrief.camera_energy
       -> directorBrief.camera_relationship
       -> directorBrief.viewer_perspective

  -> buildStage2Context(factual, shotPlan, promptSlots, {}, directorBrief, stage2Scope)
       -> compactContext.camera_style
       -> compactContext.camera_energy
       -> compactContext.camera_relationship
       -> compactContext.viewer_perspective

  -> compressStage2Assembly(...)
       -> DIRECTOR_BRIEF.camera
       -> SHOT_PLAN.plan.camera_behavior
       -> PROMPT_SLOTS.CAMERA_BEHAVIOR
       -> COMPACT_CONTEXT.camera_style
```

## Exact Duplicate Fields

### 1. `camera_style` -> `directorBrief.camera`

Producer:

```js
deriveSocialCameraIntelligence()
```

Typical values:

- `dynamic vertical creator framing`
- `intimate vertical creator framing`
- `handheld creator selfie framing`

Consumer duplication:

```js
buildDirectorBrief()
directorBrief.camera = socialCamera.camera_style || ...
```

Retained again:

```js
compressStage2Assembly()
minimalContext.camera_style = compactContext.camera_style
```

Result:

```txt
DIRECTOR_BRIEF.camera = "intimate vertical creator framing"
COMPACT_CONTEXT.camera_style = "intimate vertical creator framing"
```

Duplication score: **1.0**

Removal risk: **low-to-medium** for removing only `COMPACT_CONTEXT.camera_style`, because `DIRECTOR_BRIEF.camera` already carries the same value in modern video prompts. Risk is not zero because Gemini may be using compact context as reinforcement.

### 2. `shotPlan.camera_behavior` -> `PROMPT_SLOTS.CAMERA_BEHAVIOR`

Producer:

```js
buildShotPlan()
```

Slot transformation:

```js
buildPromptSlots()
translateToSocialMotionLanguage(rewriteSlotLanguage(shotPlan.camera_behavior, factual), factual)
```

Result:

```txt
SHOT_PLAN.plan.camera_behavior = "Close vertical social-media framing with subtle phone-camera realism"
PROMPT_SLOTS.CAMERA_BEHAVIOR = "Close vertical social-media framing with subtle phone-camera realism"
```

Duplication score: **0.85-1.0**

Removal risk: **medium-to-high**. This duplication is structural: `SHOT_PLAN` documents the plan, while `PROMPT_SLOTS` enforces ordering. Removing either may affect slot-following behavior even if the information is duplicated.

## Near-Duplicate Fields

### `camera_relationship`

Produced values:

- `viewer-facing creator framing`
- `creator-camera intimacy`

These overlap semantically with `directorBrief.camera` but carry relationship intent rather than camera mechanics. They are not retained directly in the compressed final prompt, but may be folded into:

- `directorBrief.visual_goal`
- full uncompressed `directorBrief.camera_relationship`
- master prompt generation

Duplication score: **0.2-0.6**

Removal risk: **medium** if removed from DirectorBrief entirely; **low** if removed only from compact context, because compression does not retain it in `minimalContext`.

### `camera_energy`

Produced values:

- `energetic vertical reel cadence`
- `relaxed vertical reel cadence`

This overlaps with `camera_style` through `vertical/reel` terms, but its unique contribution is cadence. It may enter `directorBrief.motion`, `visual_goal`, and master prompt wording.

Duplication score: **0.2-0.5**

Removal risk: **medium** if removed from DirectorBrief; **low** if removed from compact context, where it is already dropped by compression.

### `viewer_perspective`

Produced value:

- `intimate short-form social-video perspective`

This overlaps with social camera style and creator framing, but it contributes social-video perspective rather than shot mechanics.

Duplication score: **0.2-0.5**

Removal risk: **medium** if removed from generation intent; **low** if removed from compact context, where it is already dropped by compression.

## Estimated Prompt Payload Duplication

Typical retained camera strings:

- `DIRECTOR_BRIEF.camera`: 30-80 chars
- `COMPACT_CONTEXT.camera_style`: 30-45 chars
- `SHOT_PLAN.plan.camera_behavior`: 60-95 chars
- `PROMPT_SLOTS.CAMERA_BEHAVIOR`: 60-95 chars

Estimated duplicated retained payload per modern platform prompt:

```txt
Exact/direct duplication:
  camera_style duplicated in DIRECTOR_BRIEF + COMPACT_CONTEXT:
  ~30-45 chars

Structural duplication:
  shotPlan.camera_behavior duplicated in SHOT_PLAN + PROMPT_SLOTS:
  ~60-95 chars

Total likely camera duplication:
  ~90-140 raw text chars
  ~140-250 serialized JSON/prompt chars after keys, quotes, labels, and punctuation
```

This is smaller than the earlier PromptComponents duplication, but it is consistent and present across all modern video platforms.

## Removal Risk Assessment

| Candidate | Risk | Reason |
|---|---:|---|
| Remove `COMPACT_CONTEXT.camera_style` when it equals `DIRECTOR_BRIEF.camera` | Low-to-medium | Likely exact duplicate in modern path. Could slightly reduce reinforcement but should not remove unique camera data. |
| Remove `camera_energy`, `camera_relationship`, `viewer_perspective` from `buildStage2Context()` only | Low | They are built in compact context but not retained by `compressStage2Assembly().minimalContext`. This is mostly dead payload before compression. |
| Remove `camera_style` from `buildStage2Context()` entirely | Medium | Final prompt currently retains it in `COMPACT_CONTEXT`. Safe only if `DIRECTOR_BRIEF.camera` remains present and equal. |
| Remove `SHOT_PLAN.camera_behavior` from final Stage 2 prompt | Medium-to-high | Prompt slots depend on shot plan; slot-only structure may need both for Gemini compliance. |
| Remove `PROMPT_SLOTS.CAMERA_BEHAVIOR` | High | Slot-enforced generation requires platform-specific slot ordering. This is active prompt control. |
| Remove `generateCameraLanguage()` | High | Used by legacy paths, image prompts, shot plan fallback, director brief fallback, validation context, and camera grammar diagnostics. |
| Remove `deriveSocialCameraIntelligence()` | High | Supplies creator-native camera language and fixes over-static social-video output. |
| Stop assigning `socialCamera.camera_style` to `directorBrief.camera` | High | This would alter modern platform prompt behavior and likely regress social-video authenticity. |

## Recommended Next Simplification Candidate

The safest next candidate is **not** removing a camera intelligence system. It is removing redundant serialization:

### Candidate A: Compact Context Camera Deduplication

Only serialize `COMPACT_CONTEXT.camera_style` when it differs from `DIRECTOR_BRIEF.camera`.

Current duplicated state:

```txt
DIRECTOR_BRIEF.camera: "intimate vertical creator framing"
COMPACT_CONTEXT.camera_style: "intimate vertical creator framing"
```

Potential cleaned state:

```txt
DIRECTOR_BRIEF.camera: "intimate vertical creator framing"
COMPACT_CONTEXT: no camera_style unless unique
```

Expected impact:

- Prompt behavior risk: low-to-medium
- Payload reduction: small but consistent
- Architecture risk: low
- Best suited for a feature-flagged experiment

### Candidate B: Pre-Compression Compact Context Cleanup

`buildStage2Context()` creates these camera fields:

- `camera_energy`
- `camera_relationship`
- `viewer_perspective`

But `compressStage2Assembly()` drops them from `minimalContext`. They contribute to intermediate payload/tracing but not the compressed final Stage 2 prompt.

Expected impact:

- Final prompt impact: likely none
- Diagnostics impact: possible
- Payload reduction before compression: small
- Risk: low

### Candidate C: Shot Plan / Prompt Slot Camera Deduplication

This is tempting because `SHOT_PLAN.camera_behavior` and `PROMPT_SLOTS.CAMERA_BEHAVIOR` are often identical. However, this duplication is part of the slot enforcement architecture.

Expected impact:

- Prompt behavior risk: medium-to-high
- Payload reduction: modest
- Recommendation: audit with a feature flag before removal

## Conclusion

Camera intelligence is duplicated, but the duplication is narrower than the earlier PromptComponents chain.

The system should keep both:

- `generateCameraLanguage()` for grounded camera grammar and legacy/image support
- `deriveSocialCameraIntelligence()` for social-video camera style

The next safe simplification candidate is **serialization cleanup**, specifically:

1. Avoid retaining `COMPACT_CONTEXT.camera_style` when it exactly equals `DIRECTOR_BRIEF.camera`.
2. Consider dropping `camera_energy`, `camera_relationship`, and `viewer_perspective` from `buildStage2Context()` if they remain unused by compressed Stage 2 assembly.
3. Leave `SHOT_PLAN.camera_behavior` and `PROMPT_SLOTS.CAMERA_BEHAVIOR` intact until a slot-dedup experiment proves no loss of prompt adherence.
